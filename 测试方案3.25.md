智慧校园系统测试方案

一、项目现状摸底

这个一体化智慧校园系统后端用的是 Java 17 + Spring Boot 3.2.4，前端是 Vue3 + Vite。从仓库代码来看，后端已经实现了四个主要业务模块：

1. 认证模块（auth）：验证码、登录退出这些基础功能
2. 系统管理（sys）：用户角色权限、菜单字典、操作日志等后台管理功能
3. 教研管理（edu）：课程排课、课件管理、考勤请假、成绩申诉等教学相关功能
4. 后勤服务（svc）和仪表盘（dashboard）：公告、宿舍、报修、校园卡、图书借阅、数据看板等功能

后端 README 里提到的一些设计亮点需要重点关注，比如用 @LogRecord 做的操作审计、Redis 缓存签到机制、成绩防篡改、文件上传安全校验、Sa-Token 的权限控制等。

前端目前只是个 Vite 工程模板，package.json 里只有 dev/build/preview 三个脚本，没有看到 vitest、jest 或 cypress 这些测试框架的配置，自动化测试这块还没搞起来。

二、现有的单元测试情况

后端在 campus-platform-backend/campus-server/src/test/java/.../modules/svc/controller/ 路径下写了 6 个控制器测试类，用的 JUnit5 + Mockito：

DashboardControllerTest：测了仪表盘缓存命中时不会重复调服务层，以及保存快照时如果已有记录会更新而不是插入重复数据。

CampusNoticeControllerTest：验证了公告列表会根据当前用户角色过滤掉不该看的公告。

CampusRepairControllerTest：确认报修的验收确认操作只有申请人自己能做，其他人操作会抛异常且数据不会变。

CampusDormControllerTest：宿舍分配时如果床位已被占用会拒绝请求；退宿不是直接删数据，而是更新状态、记录签出时间、同步房间占用数。

CampusCardControllerTest：挂失接口会在保存前自动填充挂失时间、学生ID、状态等字段；充值接口会拒绝非法的手动本地充值请求。

CampusBookControllerTest：借书时如果库存扣减失败会拒绝请求且不会生成借阅记录；还书时如果不是本人借阅会拒绝操作。

总的来说，现有的测试主要覆盖了 svc 模块控制器的参数校验、权限校验和状态流转逻辑，sys 和 edu 模块还没有同等覆盖率的单元测试，集成测试和端到端测试更是没有。

三、本轮测试目标

分三层来保障质量：

单元测试层：把控制器和服务层的关键分支都覆盖到，包括参数校验、权限判断、状态机流转、缓存和异常处理等场景。

集成测试层：验证跟 MySQL、Redis 的交互是否正确，事务一致性、缓存穿透和 TTL 行为、数据持久化和查询都要测到。

系统/端到端测试层：前后端联调链路通不通，核心业务场景能不能跑通，比如登录流程、不同角色看到的页面和接口是否对、典型业务流程是否闭环。

四、测试范围和用例清单

根据模块划分，建议重点验收以下场景：

4.1 认证与权限（auth / sys 公共部分）

验证码获取和校验，成功和失败的情况都要测
登录成功和失败的场景（账号错误、密码错误、验证码错误）
退出登录后 token 是否失效
RBAC 权限控制：不同角色访问同一个接口，有的能进有的不能进，验证 @SaCheckPermission 和 @SaCheckRole 是否生效

4.2 系统管理（sys）

用户的增删改查、状态切换、密码重置
角色的增删改查，分配权限菜单后是否生效
菜单的树形结构查询是否正常
字典类型和字典数据的管理
操作日志和登录日志的分页查询

4.3 教研管理（edu）

课程的增删改查，教师和班级的绑定关系
排课管理和课表查询，数据是否一致
课件上传的 MD5 去重、删除同步、下载计数
评价功能：每人限评一次的规则是否生效
考勤功能：Redis 签到的 6 位随机码和 TTL 过期行为，统计是否正确
请假审批：通过和驳回的状态流转
成绩录入：归档后防篡改，申诉解锁流程

4.4 后勤服务（svc）与仪表盘（dashboard）

公告：草稿和发布状态切换，已读标记自动落库和幂等性
宿舍：房间床位计数同步，分配和退宿状态正确性，占用边界条件
报修：提交→受理→完成→验收的完整流程，非申请人操作拒绝
校园卡：消费记录查询，充值时禁用非法入口，挂失和解挂的状态机
图书：借书时库存扣减一致性，还书逾期计算（如果做了的话），非本人借阅拒绝
仪表盘：概览数据缓存命中和失效场景，保存快照时更新旧记录避免重复键

五、怎么执行测试

5.1 后端单元测试

进后端模块直接跑：

cd campus-platform-backend/campus-server
mvn test

现在的单测主要是用 Mockito 做控制器逻辑验证，可以作为最基础的回归基线。

5.2 前端测试

前端目前没有自动化测试脚本，只能先手工测。启动命令：

cd campus-platform-frontend
npm install
npm run dev

然后手工联调和回归验证。

六、目前的风险和缺口

自动化测试覆盖面不够：只有 svc 模块有控制器单测，sys 和 edu 模块缺少同等级别的单元测试。
集成测试缺失：没有看到跟 MySQL、Redis 的集成测试，也没有 Testcontainers 配置，后续需要补环境和用例。
前端自动化测试缺失：现在只是个模板工程，建议后面补上 vitest 做组件和函数测试，再用 cypress 或 playwright 做端到端测试。
