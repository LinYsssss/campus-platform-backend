三、发现的问题

虽然单元测试都过了，但编译和测试体系还是暴露了一些问题：

3.1 构建参数有警告，影响可移植性

javac 报了个警告："未与 -source 17 一起设置系统模块位置"。建议改成用 --release 17 来规范编译，避免不同 JDK 环境下出现模块或运行差异。

根项目的 pom.xml 现在用的是 maven-compiler-plugin 的 source/target 配置方式，没用到 --release。

3.2 测试代码里有不安全操作警告

编译 CampusNoticeControllerTest 的时候报了"使用了未经检查或不安全的操作"。建议定位一下具体是哪里的问题，比如原始泛型、强制类型转换或者 Mockito 返回类型不匹配，然后补上泛型约束或者加 @SuppressWarnings 并写清楚原因，别让这种警告一直积累。

3.3 测试覆盖面还是不够，只有 svc 域

现在能看到的自动化测试主要集中在 svc 域的控制器。sys 和 edu 域的控制器和服务层测试还没跟上，导致权限校验、Redis 签到、成绩防篡改、文件上传安全这些关键业务规则缺少自动回归保障。

3.4 缺集成测试和端到端测试，有"逻辑对但数据层不对"的风险

现在的测试基本都是 Mockito 单元测试，测不了 MySQL 和 Redis 的真实交互、事务边界、缓存 TTL、还有 SQL 和 Mapper 的实际行为。

建议引入集成测试，可以用 Testcontainers 或者本地环境，把核心链路都覆盖一下：登录流程、公告已读、报修状态流转、借还书库存一致性这些。

3.5 前端自动化测试体系完全没建

前端现在就是个 Vite 模板脚手架，package.json 里没配 vitest、jest、cypress、playwright 这些测试脚本。

建议后面补齐前端单元测试和组件测试，再补一些必要的 E2E 测试，把关键页面和权限展示逻辑覆盖到。

四、结论

这次跑的后端单元测试都通过了（10 个用例全绿），但项目在构建警告治理、测试覆盖广度、集成和端到端测试体系、前端测试体系这几块还有明显短板。建议拿这份总结作为后面补测和工程化改进的依据。

五、后续改进建议（按优先级排）

1. 先把构建警告处理了，把 source/target 改成 --release 或者等效的规范配置。
2. 修复测试编译告警，至少做到这个模块没有 unchecked 警告。
3. 扩展测试用例到 sys 和 edu 域，覆盖控制器和服务层的关键分支。
4. 增加集成测试，覆盖 MySQL、Redis、Mapper 和缓存 TTL、幂等性这些场景。
5. 给前端补齐自动化测试，包括组件/函数测试和关键 E2E 场景。
